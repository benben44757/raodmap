Section 01

Project Overview

What problem does it solve?

When you talk to an AI chatbot, it replies in plain English. That's great for conversation, but computers can't execute plain English — they need precise, structured instructions. This project teaches a small AI model to reply not with words, but with a structured function call: the exact function name and arguments a program needs to act on the request.

💬 Normal AI reply

User asks: "What is the sum of 40 and 2?"

"The sum of 40 and 2 is 42."

A human can read this, but a program can't use it directly.

⚙️ Function calling reply

Same question, structured output:

{
  "function": "fn_add_numbers",
  "arguments": { "a": 40, "b": 2 }
}

Any program can parse and execute this immediately.

What are the inputs?

📄 function_calling_tests.json INPUT

A list of natural-language prompts. Think of it as the list of questions users might ask.

[
  {"prompt": "What is the sum of 2 and 3?"},
  {"prompt": "Greet shrek"},
  {"prompt": "Reverse 'hello'"}
]
📋 functions_definition.json SCHEMA

A catalog of available functions the LLM can choose from, with their argument types.

[
  {
    "name": "fn_add_numbers",
    "description": "Add two numbers",
    "parameters": {
      "a": {"type": "number"},
      "b": {"type": "number"}
    }
  }
]

What is the output?

✅ function_calling_results.json OUTPUT

For every prompt, a JSON object with the chosen function name and the extracted arguments:

[
  {
    "prompt": "What is the sum of 2 and 3?",
    "name": "fn_add_numbers",
    "parameters": {"a": 2.0, "b": 3.0}
  },
  {
    "prompt": "Reverse the string 'hello'",
    "name": "fn_reverse_string",
    "parameters": {"s": "hello"}
  }
]

What is "function calling"?

Function calling is the skill of making an LLM identify the right function to execute from a prompt, and extract the correct arguments from the text. The model doesn't actually run the function — it just decides which one to call and with what values. Think of it like a smart dispatcher who reads a request and fills out the right form.

What is "constrained decoding" and why is it required?

The core challenge

Language models are unpredictable. A small 0.6B model asked to produce JSON might succeed only 30% of the time. It might add extra text, forget quotes, or produce invalid JSON.

Constrained decoding solves this by controlling which tokens the model is even allowed to generate, step by step. Instead of hoping the model produces valid JSON, you mathematically guarantee it by setting the probability of all invalid tokens to zero at every step.

With constrained decoding, even a tiny 0.6B model achieves 99%+ reliability. This is the main skill this project teaches.

The Constrained Decoder State Machine

The decoder needs to know what position it is in inside the JSON structure at each step. This is the key insight of the project:

State machine for generating: {"name": "...", "parameters": {...}} START → must emit: { AFTER_{ → must emit: "name" AFTER_"name" → must emit: : IN_NAME_VALUE → must emit one of the valid function names e.g. "fn_add_numbers" or "fn_greet" AFTER_NAME → must emit: , IN_PARAMS → must emit param key, then :, then typed value if param type is "number": only digits/decimal tokens valid if param type is "string": build string token by token END → emit: }
Section 03

Beginner-Friendly Explanations

📦 JSON

What it is
JSON (JavaScript Object Notation) is a standard text format for representing structured data. It uses keys and values, arrays, and nested objects. It looks like a Python dictionary.
Why you need it
All your input files and output files are JSON. The LLM must generate valid JSON. Python's json module lets you read and write it.
Simple example
import json

# Reading JSON from a file
with open("data.json") as f:
    data = json.load(f)     # dict or list

# Writing JSON to a file
with open("output.json", "w") as f:
    json.dump(result, f, indent=2)

# JSON string ↔ Python object
text = '{"name": "Alice", "age": 30}'
obj = json.loads(text)      # → dict
back = json.dumps(obj)      # → string

🏗️ Python Classes

What it is
A class is a blueprint for creating objects. An object bundles data (attributes) and behavior (methods) together. You use classes to organize complex logic.
Why you need it
Your ConstrainedDecoder needs to hold a model, a vocabulary, and a state — a class is the natural way to organize this. Pydantic also uses classes for validation.
Simple example
class Decoder:
    def __init__(self, vocab: dict):
        self.vocab = vocab          # stored data
        self.generated: list = []   # state

    def step(self, logits: list) -> int:
        # find the best valid token
        best = max(range(len(logits)),
                   key=lambda i: logits[i])
        self.generated.append(best)
        return best

d = Decoder(vocab={"0": "{"})
d.step([0.1, 0.9, 0.3])

📝 Type Hints

What it is
Type hints are annotations that tell Python (and tools like mypy) what type each variable or function argument should be. They don't enforce anything at runtime — they're documentation + static checking.
Why you need it
The project requires type hints on all functions and mypy must pass without errors. They also make your code much easier to understand.
Simple example
from typing import List, Dict, Optional, Any

def get_valid_tokens(
    logits: List[float],
    allowed: List[int]
) -> Dict[int, float]:
    return {i: logits[i] for i in allowed}

def load_file(path: str) -> Optional[dict]:
    try:
        with open(path) as f:
            return json.load(f)
    except FileNotFoundError:
        return None

🛡️ Pydantic

What it is
Pydantic is a library that uses Python classes to validate data. You define a model (a class inheriting from BaseModel), and when you create an instance, Pydantic automatically checks that all fields have the right types and values.
Why you need it
The project requires all classes to use Pydantic. It protects you from bad input data — if a JSON file is missing a field or has the wrong type, Pydantic raises a clear error instead of crashing mysteriously later.
Simple example
from pydantic import BaseModel

class FunctionDefinition(BaseModel):
    name: str
    description: str
    parameters: dict[str, dict]

# Valid data — works fine
fn = FunctionDefinition(
    name="fn_add",
    description="Add numbers",
    parameters={"a": {"type": "number"}}
)

# Invalid data — Pydantic raises ValidationError
fn2 = FunctionDefinition(
    name=123,     # ← should be str!
    description="test",
    parameters={}
)

🔤 Tokenization

What it is
Language models don't read text character by character or word by word. They read tokens — chunks of text that are somewhere between a character and a word. Common words are one token; rare words are split into multiple tokens.
Why you need it
The LLM SDK's encode(text) converts your prompt string into a list of token IDs. You pass these IDs to get_logits_from_input_ids(). You also need tokens to know which ones form valid JSON characters.
Simple example
# "Hello world" might tokenize to:
# ["Hello", "Ġworld"]  (Ġ = preceding space)
# with IDs: [9906, 1917]

# A JSON brace "{" might be token ID 90
# A digit "2" might be token ID 17
# "fn_add" might be split into: ["fn", "_add"]

# The vocab file maps:
# 90 → "{"
# 17 → "2"
# ...
# You need this map to know: at this step,
# which tokens form valid JSON continuations?

📚 Vocabulary Files

What it is
The vocabulary file (usually a JSON file) is a complete list of every token the model knows, indexed by their ID. It's like a dictionary: token ID → token string.
Why you need it
Constrained decoding needs this map. At each step, you look at all ~32,000 possible tokens (the vocab size), and you need to know their string representation to decide which ones are valid JSON continuations.
Simple example
import json

# The vocab file looks like:
# {"Ġthe": 279, "of": 315, "{": 517, ...}
# or as a list indexed by token ID

# Load it
vocab_path = model.get_path_to_vocab_file()
with open(vocab_path) as f:
    vocab = json.load(f)

# Build reverse map: id → string
# e.g., {517: "{", 318: "\"", ...}
id_to_token = {v: k for k, v in vocab.items()}

📊 Logits

What it is
Logits are raw unnormalized scores the model outputs for each token in its vocabulary. Higher logit = higher probability for that token to be next. The vocabulary has ~32,000 tokens, so there are ~32,000 logits at each step.
Why you need it
You call get_logits_from_input_ids(input_ids) to get these scores. Then you mask invalid ones (set to −∞) and pick the highest remaining one. This is the entire mechanism of constrained decoding.
Simple example
import math

logits = model.get_logits_from_input_ids(input_ids)
# → [0.01, -2.3, 8.7, 0.002, ...]
#    tok0   tok1  tok2  tok3

# If at this position only "{" is valid (token 517):
NEG_INF = float('-inf')
masked = [NEG_INF] * len(logits)
masked[517] = logits[517]  # only allow "{"

# Pick the best
next_token = masked.index(max(masked))
# next_token = 517

🔒 Constrained Decoding — The Heart of the Project

What it is
Constrained decoding is the process of generating text token by token while, at each step, restricting which tokens are allowed based on the structure you want to produce. Invalid tokens get their logits set to −∞, so they can never be selected.
Why you need it
Without it, the model sometimes outputs text like Here is the function call: {...} instead of raw JSON. Or it forgets a closing brace. Or it uses the wrong type. With constrained decoding, structural validity is mathematically guaranteed.
The loop (conceptually)
def generate_json(model, vocab, prompt, schema):
    input_ids = model.encode(prompt)
    output_ids = []
    state = "START"   # tracks where we are in JSON

    while not is_done(state):
        logits = model.get_logits_from_input_ids(
            input_ids + output_ids
        )

        # Key step: compute which tokens are valid
        valid_tokens = get_valid_tokens(state, schema, vocab)

        # Mask everything else to -inf
        for i in range(len(logits)):
            if i not in valid_tokens:
                logits[i] = float('-inf')

        # Pick best token
        next_id = logits.index(max(logits))
        output_ids.append(next_id)

        # Update state machine
        state = advance_state(state, vocab[next_id])

    return decode_to_string(output_ids)

Section 07

Risk Analysis

🚨 Skipping constrained decoding and just prompting the model

The spec explicitly says: "Your solution must NOT rely on the model spontaneously producing correct JSON from a prompt." If you only prompt the model and hope for the best, you will fail the evaluation — even if it works most of the time. The evaluator will ask to see your constrained decoding code.

Fix: Implement the full generation loop with logit masking from day one. This is non-negotiable.

🚨 Using dspy, transformers, or outlines

These libraries are completely forbidden. Even if they solve the problem perfectly, using them will result in a failing grade. The goal is to implement constrained decoding from scratch using only the provided SDK.

Fix: Only use: numpy, pydantic, json, the llm_sdk package. That's it.

⚠️ Misunderstanding the vocab file format

The vocab file might be a dict of token→id, a list, or have special encoding for spaces (the "Ġ" prefix for space-prefixed tokens). Getting this wrong means your token masking won't work at all.

Fix: On Day 7, spend real time examining the vocab file before writing any code. Print the first 20 entries and understand the format.

⚠️ State machine bugs producing invalid JSON

The most common bug is a state transition that's off by one — for example, allowing a "," before the last parameter's closing brace. This produces syntactically invalid JSON that fails json.loads().

Fix: Add a debug mode that prints each token as it's generated. Unit test every state transition independently before running the full pipeline.

⚠️ Infinite loops during generation

If your state machine gets stuck (e.g., in a string-generating state that never finds a closing quote token), the generation loop runs forever.

Fix: Add a maximum token limit (e.g., 200 tokens). If the limit is reached, raise an error with a clear message.

⚠️ Hardcoding solutions for the test prompts

The spec warns explicitly: "Do not hardcode solutions based on the provided examples." The evaluator will use different prompts and possibly different functions. Your system must generalize.

Fix: Test with custom prompts you invent yourself. Change the functions_definition.json to something new and verify your system adapts.

Section 08

Evaluation Preparation

An evaluator will ask you to explain and demonstrate your code. Be ready for all of these.

What is constrained decoding and how did you implement it?
Ideal answer: Constrained decoding is a technique where, at each step of LLM token generation, I compute which tokens are valid given the current generation state (what JSON structure I'm building), and set all other token logits to negative infinity so they can never be selected. I implemented a state machine that tracks whether I'm generating the opening brace, the function name, a colon, a parameter key, a parameter value of a specific type, or a closing brace. At each state, a different set of token IDs is allowed. This guarantees that every token the model generates is both valid JSON and matches the function schema.
Key concept: logit masking + state machine + guaranteed structural validity
Why can't you just prompt the model to produce JSON?
Ideal answer: Prompting alone is unreliable with small models. A 0.6B parameter model might produce valid JSON only 30% of the time — it might add extra text, use wrong quotes, forget brackets, or produce numbers as strings. With constrained decoding, validity is mathematically guaranteed because invalid tokens literally cannot be selected. The model's "opinion" about what to generate next is guided by its knowledge of the task, but the structural validity is enforced by us, not by the model.
Key concept: the separation between model intelligence (choosing WHICH function) and structural enforcement (HOW the output is shaped)
How do you determine which tokens are valid at each step?
Ideal answer: I use the vocabulary file, which maps every token ID to its string representation. At each state, I iterate over all ~32,000 tokens and check if their string representation is consistent with the current expected output. For example, when generating a number parameter value, I allow tokens whose strings are digits (0-9), a minus sign, or a decimal point. I also check that the token doesn't break the partially-built output — for example, I don't allow a second decimal point if one already exists in the number being built.
Key concept: vocab file usage, prefix/suffix checking, accumulated state
How did you handle the function selection? Does the LLM choose it?
Ideal answer: Yes, the LLM genuinely chooses the function, not a heuristic. When generating the "name" field, I constrain the valid tokens to those that are prefixes of one of the known function names. The model's logits still determine which function name gets generated — if the model assigns higher probability to tokens leading to "fn_add_numbers" than "fn_greet", that function wins. The constraint just ensures the output is a valid function name from our list, not random text.
Key concept: prefix-constrained generation, LLM semantics preserved within structural constraints
How does your program handle malformed input files?
Ideal answer: Every file I/O operation is wrapped in a try/except block. If the file doesn't exist, I print a clear error message and exit with a non-zero code. If the file exists but contains invalid JSON (json.JSONDecodeError), I catch that and inform the user. If the JSON structure doesn't match the expected schema, Pydantic's ValidationError is caught and displayed. The program never crashes unexpectedly — it always terminates gracefully with an informative message.
Key concept: defensive programming, Pydantic validation, graceful error handling
What is Pydantic and why did you use it?
Ideal answer: Pydantic is a data validation library that lets me define data schemas as Python classes. When I create an instance of a Pydantic model, it automatically validates that all fields have the correct types. I used it because the spec requires it, but also because it's genuinely useful — it catches malformed input files early and gives clear error messages. Without it, a missing field in the JSON input might cause a confusing KeyError deep inside my generation code.
Key concept: data validation, type safety, early failure
Walk me through what happens when I run your program with a prompt.
Ideal answer: 1. The CLI parses arguments and loads both JSON files using the loader. 2. The loader validates them with Pydantic models. 3. For each prompt, the prompt builder creates a text string that describes the available functions and asks for a function call. 4. The LLM SDK's encode() converts this text to token IDs. 5. The constrained decoder runs a loop: at each step it gets logits from the model, masks invalid token logits to -inf based on the current state, and selects the highest remaining token. 6. The loop terminates when the closing brace is generated. 7. The resulting token IDs are decoded to a JSON string, parsed, and validated. 8. All results are collected and written to the output file by the writer.
Key concept: end-to-end pipeline understanding
